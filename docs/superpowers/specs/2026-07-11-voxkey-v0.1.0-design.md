# VoxKey v0.1.0 Design

**Status:** Approved product direction; implementation planning pending review

## Objective

Ship VoxKey as a separate, public, local-first Windows dictation application that feels fast and invisible until needed. It will be released from a new `Deepak-80085/voxkey` repository under the MIT license.

VoxKey is English-only. It records while the user holds Right Ctrl, transcribes with the local `small.en` faster-whisper model, polishes with local Ollama `qwen3.5:0.8b`, restores the original target window, and pastes only the polished result.

## Product principles

- No account, cloud service, API key, or raw-transcript paste fallback.
- Right Ctrl works globally wherever Windows permits input hooks and focus/paste access.
- A normal process is used by default. When the target is elevated, VoxKey explains that it must be run as administrator to interact with that app. Secure desktop surfaces are unsupported.
- The product remains visually absent while idle, except for its tray icon.
- Recording and processing are communicated with a premium, Siri-inspired HUD and subtle sound cues.
- Inference remains local and uses `small.en`; `base.en` is never selected or downloaded.

## Scope

### v0.1.0 — polished foundation

1. **Native desktop shell**
   - Replace the Tkinter control center with a PySide6/Qt Windows shell.
   - Run with a system tray icon and no persistent desktop window.
   - Add tray actions: Open settings, toggle sounds, model status/repair, open diagnostics folder, Quit.
   - Closing settings hides it to the tray. Quit shuts down hooks, UI, and mutex cleanly. This differs from the old control-center close behavior, which exited unexpectedly.

2. **Transient dictation HUD**
   - A transparent, always-on-top, mouse-transparent bottom-center overlay.
   - It never claims keyboard focus or activates itself.
   - It appears only from valid Right Ctrl press through processing completion/failure.
   - State contract:
     - `Listening`: animated multicolor voice orb, live audio waveform, “Listening”, and release hint.
     - `Processing`: quieter compressed/pulsing orb, “Transcribing…” then “Polishing…”.
     - `Complete`: short confirmation glow, then fade after successful paste.
     - `Error`: restrained error color and readable message, then fades; the detailed cause stays available in settings and diagnostics.
   - Animation and fade timing must not delay audio capture, decoding, target restoration, or paste.

3. **Sound feedback**
   - Enabled by default.
   - Short locally bundled or generated non-verbal cues: recording start, successful paste, and actionable error.
   - A tray/settings mute toggle persists locally.
   - Sound playback failures never affect dictation.

4. **Settings and repair experience**
   - A compact, polished settings window containing microphone selection, Right Ctrl hotkey explanation, sound toggle, speech model device/health, Ollama/writer health, repair controls, and diagnostic file location.
   - The first release exposes the fixed Right Ctrl hotkey rather than promising unsupported arbitrary remapping.
   - Model repair is explicit and shows determinate/indeterminate progress and actionable errors.

5. **Production reliability**
   - Retain existing single-instance protection, lifecycle logs, captured target HWND, real-device audio, GPU discovery via Ollama CUDA libraries, same-model CPU fallback, and strict local writer behavior.
   - Add structured timings for recording, transcription, polishing, focus restoration, and paste.
   - Surface GPU/CPU device and model readiness in the UI and log.
   - Preserve the latest recording diagnostic only according to the existing local-data policy; explain its location in diagnostics.

6. **Open-source release package**
   - Extract the VoxKey worktree to a fresh Git repository, not the SimpleSpeech repository.
   - Add MIT `LICENSE`, security/privacy statement, contribution guidance, architecture diagram, troubleshooting guide, screenshot/GIF assets, versioned changelog, and explicit unsigned-installer notice.
   - CI validates lint/unit tests, packaging smoke checks, and installer artifact creation.
   - Publish GitHub release `v0.1.0` with a Windows installer and SHA-256 checksum. Code signing is not a prerequisite, but unsigned status must be plainly documented.

### v0.2.0 — follow-on power features

- First-run onboarding.
- Start with Windows.
- Local dictation history with clear retention/deletion controls.
- Vocabulary editor.
- Per-application behavior and advanced workflow controls.

These features are excluded from v0.1.0 except where the v0.1 architecture needs a stable extension point.

## Architecture

### UI boundary

A Qt application shell owns the tray, settings window, HUD, sound feedback, UI state rendering, and orderly exit. It consumes state updates from the existing runtime/controller through a thread-safe event interface. The UI does not call inference code directly.

### Dictation boundary

The existing hotkey/audio service remains responsible for globally observing Right Ctrl, retaining the pre-dictation target HWND, capture start/stop, and dispatching work. It emits lifecycle events rather than manipulating UI widgets.

### Pipeline boundary

The controller coordinates speech-model readiness, local transcription, required local writing polish, focus restoration, and paste. It reports explicit stages and elapsed timings. A pipeline error emits an error event and never pastes raw text.

### Runtime boundary

Runtime manages `%LOCALAPPDATA%\VoxKey`, preferences, log paths, retained recording path, model cache, single-instance mutex, and clean shutdown. The app must release the mutex only once all background services are stopped.

## User flow

1. User starts VoxKey; it validates local models and appears in the tray as Ready or Needs repair.
2. User clicks in any ordinary target app and holds Right Ctrl.
3. VoxKey captures the target HWND, plays the start cue, and shows the Listening HUD without taking focus.
4. On release, it stops capture and switches the HUD to Processing.
5. It performs local GPU `small.en` transcription where available, then local Ollama polishing.
6. It restores the captured target window, pastes the polished text, plays the complete cue, and fades the HUD.
7. On an error, it plays the error cue, displays the concise cause, logs timing/context, and does not paste.

## Error handling

- Missing/corrupt speech model: `Needs repair` with an explicit repair action; no dictation starts.
- Ollama unavailable/model missing/writer timeout: error HUD and repair guidance; no raw text paste.
- GPU unavailable: transparently attempt CPU with the same `small.en` model and report the selected device.
- Elevated target: report access limitation and corrective instruction.
- Multiple launch: raise/show the existing instance if feasible; otherwise terminate the new process without microphone contention.
- UI/audio/sound errors are isolated so they cannot crash the hotkey service.

## Test and acceptance strategy

- Unit tests: state-to-HUD mapping, tray action wiring, persisted sound setting, shutdown/mutex behavior, timing event schema, and no-focus HUD properties.
- Integration tests: injected hotkey lifecycle drives UI events; pipeline success/error drives the correct HUD/sound states; settings repair invokes controller repair.
- Windows smoke tests: installer install/upgrade/uninstall; launch then quit then relaunch; tray menu operation; Windows Notepad real paste acceptance; elevated-target warning where testable; GPU device evidence in packaged-app log.
- Performance acceptance: on the GTX 1650, record timings for an ordinary short utterance and retain a regression benchmark. The HUD must appear immediately on key press and must not increase the established inference time.

## Release acceptance criteria

- `VoxKey.exe` starts once, exposes Ready/Needs repair state, and can quit/relaunch cleanly.
- Hold Right Ctrl gives visible/sound listening feedback without stealing focus.
- Successful local transcription + polish restores focus and pastes exact polished text in real Notepad.
- Errors never paste raw transcription.
- Tray, settings, mute persistence, model health and repair are usable.
- The new repository has MIT licensing, build/release instructions, local-only privacy claim substantiated by architecture, CI, and a release installer checksum.
