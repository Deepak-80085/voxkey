# VoxKey V2 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace SimpleSpeech with VoxKey, an English-only local dictation app that requires validated `small.en` transcription and validated local Ollama polishing before it enables one global dictation hotkey.

**Architecture:** Move the current startup/model logic into focused modules: `voxkey_runtime` owns data paths and state, `speech_models` owns `small.en` validation and GPU-to-CPU loading, `writing_model` owns strict local polishing, and `app` coordinates UI/hotkey state. A Tkinter control-center and setup flow show health and repair actions; the background service runs only in `Ready` state.

**Tech Stack:** Python 3.12, Tkinter, faster-whisper, Hugging Face Hub, Ollama HTTP API, pynput, sounddevice, pystray, Pillow, PyInstaller, Inno Setup, unittest.

## Global Constraints

- English-only; use `small.en` as the required default transcription model.
- Do not download, configure, or fall back to `base.en`.
- Require local writing-model polishing; never paste raw/unpolished transcription.
- Keep audio/text processing local after initial model setup; no accounts, keys, hosted APIs, or cloud transcription.
- CPU-first: GPU acceleration is optional and must retry the same `small.en` model on CPU after GPU failure.
- Model errors transition to a recoverable `Needs repair` state, not an unhandled exception dialog.
- Store VoxKey-owned settings, model metadata, logs, vocabulary, and recordings in `%LOCALAPPDATA%\VoxKey`.
- Do not alter untracked `index.html`.
